## 1. Resumen del cambio

El PR agrega el endpoint `POST /accounts/{address}/graduate-to-self-custody` que:

1. Recibe una `userPublicKey`.
2. Envía una transacción on-chain que **añade la clave del usuario y revoca las claves custodiales en una sola transacción** (`GraduateAccountTransaction`).
3. Marca la cuenta como `non-custodial` en la base de datos.
4. Bloquea posteriores firmas/transacciones API sobre cuentas ya graduadas (`403 Forbidden`).

El fix específico al bug reportado está en `accounts/service.go`: después de que la transacción on-chain tiene éxito, si `SaveAccount` falla, intenta un **rollback on-chain** restaurando las claves custodiales y revocando la clave graduada.

## 2. ¿Qué cambió exactamente en `accounts/service.go`?

Archivo: `accounts/service.go` (rama `tasks/W-05-graduate-to-self-custody`)

### 2.1 Antes del fix (`81fcb66` / sin rollback)

```go
_, _, err = s.txs.Create(ctx, true, account.Address, code, args, transactions.General)
if err != nil {
    return nil, err
}

account.Type = AccountTypeNonCustodial
if err := s.store.SaveAccount(&account); err != nil {
    return nil, err
}

return &account, nil
```

Problema: si `s.txs.Create(..., sync=true)` tenía éxito on-chain pero `SaveAccount` fallaba, la cuenta quedaba **graduada on-chain pero marcada como custodial en la DB**. Eso dejaba la base de datos inconsistente con la realidad on-chain.

### 2.2 Después del fix (`a547a6e`)

Se añadieron tres cosas en `accounts/service.go`:

1. **Backup de claves custodiales** antes de la transacción (l. 189-192):

```go
custodialPublicKeys := make([]string, len(account.Keys))
for i, key := range account.Keys {
    custodialPublicKeys[i] = key.PublicKey
}
```

2. **Intento de rollback on-chain** si `SaveAccount` falla (l. 226-232):

```go
account.Type = AccountTypeNonCustodial
if err := s.store.SaveAccount(&account); err != nil {
    rollbackErr := s.restoreCustodialAccount(ctx, account.Address, custodialPublicKeys, account.Keys[len(account.Keys)-1].Index+1)
    if rollbackErr != nil {
        return nil, fmt.Errorf("save graduated account: %w; rollback failed: %v", err, rollbackErr)
    }
    return nil, fmt.Errorf("save graduated account: %w", err)
}
```

3. **Nueva función `restoreCustodialAccount`** (l. 352-377):

```go
func (s *ServiceImpl) restoreCustodialAccount(ctx context.Context, address string, restorePublicKeys []string, graduatedKeyIndex int) error {
    if len(restorePublicKeys) == 0 {
        return nil
    }

    restoreKeyValues := make([]cadence.Value, 0, len(restorePublicKeys))
    for _, publicKey := range restorePublicKeys {
        restoreKeyValues = append(restoreKeyValues, cadence.String(publicKey))
    }

    signedTx, err := s.txs.Sign(
        ctx,
        address,
        template_strings.RestoreCustodialAccountTransaction,
        []transactions.Argument{
            cadence.NewArray(restoreKeyValues),
            cadence.NewInt(graduatedKeyIndex),
        },
    )
    if err != nil {
        return err
    }

    _, err = flow_helpers.SendAndWait(ctx, s.fc, signedTx.Transaction, s.cfg.TransactionTimeout)
    return err
}
```

La transacción de rollback (`RestoreCustodialAccountTransaction`, en `templates/template_strings/transactions.go` l. 153-171) hace:

- Re-agrega las claves custodiales a la cuenta.
- Revoca la clave graduada del usuario (`graduatedKeyIndex`).

### 2.3 Otras adiciones en `accounts/service.go`

- `RequireCustodialForSigning` (l. 237-239): delega a `accounts/signing_guard.go` para rechazar firmas de cuentas no-custodiales.
- `normalizeUserPublicKey` (l. 332-350): valida formato hex y algoritmo `ECDSA_P256`.
- `SyncAccountKeyCount` (l. 386-388): ahora también bloquea si la cuenta es no-custodial.

## 3. GraduateToSelfCustody: ¿qué pasa si la tx on-chain tiene éxito pero `SaveAccount` falla?

### 3.1 Comportamiento implementado

El servicio **intenta un rollback on-chain** con `restoreCustodialAccount`. El resultado posible es:

- **Rollback exitoso:** devuelve solo el error original de `SaveAccount`.
- **Rollback fallido:** devuelve un error compuesto: `save graduated account: <db err>; rollback failed: <rollback err>`.
- **No hay reintentos** ni cola de compensación. Es un único intento sincrónico dentro del request HTTP.

### 3.2 Cobertura de tests

Existe un test unitario con mocks: `accounts/service_graduate_test.go::TestGraduateToSelfCustodyRollsBackOnSaveFailure`.

Este test verifica que:

- Se envía exactamente una transacción de graduación.
- Se intenta firmar exactamente una transacción de rollback.
- El error final contiene el error de DB.

**Sin embargo, es un test de mocks.** No ejecuta la lógica real de Flow, no valida que el rollback pueda firmar como la cuenta graduada, ni que la red acepte la transacción de restauración.

### 3.3 Problema crítico: el rollback es técnicamente inviable si la graduación tuvo éxito

La transacción `GraduateAccountTransaction` es atómica on-chain: en una sola ejecución añade la clave del usuario **y revoca todas las claves custodiales**. Una vez que se confirma:

- La cuenta on-chain **solo tiene activa la clave del usuario**.
- El servicio **no posee la clave privada del usuario**.

Para enviar `RestoreCustodialAccountTransaction`, el servicio debe firmar como *proposer* de la cuenta. `restoreCustodialAccount` llama a `s.txs.Sign(...)`, que internamente:

1. Ejecuta el `custodialSigningGuard` leyendo `account.Type` de la DB. Como `SaveAccount` falló, la DB todavía dice `custodial`, por lo que **el guard no bloquea**.
2. Llama a `keys.Manager.UserAuthorizer(...)`, que lee de la DB una clave custodial para firmar.
3. Firma y envía la transacción con esa clave custodial.

El problema: **esa clave custodial ya fue revocada on-chain** por la transacción de graduación. La red rechazará la transacción de rollback.

Por lo tanto, en el escenario real de fallo de DB post-graduación, el rollback **probablemente falle**. Quedaría:

- On-chain: cuenta graduada, sin claves custodiales activas.
- DB: cuenta todavía marcada como `custodial`.

Esto es una inconsistencia grave: la API seguiría permitiendo (o intentando) firmar como custodial sobre una cuenta que on-chain ya no lo es.

### 3.4 Otros riesgos

- **Muerte del proceso:** si el servicio se reinicia/crashea entre `s.txs.Create(..., sync=true)` y `SaveAccount`, no hay ningún job ni proceso de reconciliación que detecte y repare la inconsistencia.
- **Concurrencia:** no hay lock a nivel de cuenta. Dos requests paralelos de graduate podrían ejecutar ambas transacciones on-chain y dejar la cuenta en un estado imposible de reconciliar.
- **No hay reintento de `SaveAccount`:** un error transitorio de DB no se reintenta; inmediatamente se intenta el rollback on-chain.

### 3.5 Recomendación

El rollback on-chain actual da una falsa sensación de seguridad. Opciones más robustas:

1. **Invertir el orden:** guardar primero un estado intermedio/pending en DB, luego ejecutar la transacción on-chain, y finalmente marcar como `non-custodial`. Si algo falla, la DB queda pendiente y se puede reconciliar posteriormente.
2. **Job de compensación asíncrono:** si `SaveAccount` falla, encolar un job que intente restaurar. Pero esto sigue sin resolver el problema de firma si las custodiales ya fueron revocadas.
3. **Reconciliador periódico:** un proceso que compare `account.Type` en DB con las claves reales on-chain y marque como `non-custodial` si detecta que la cuenta ya está graduada.
4. **Aceptar la irreversibilidad on-chain:** documentar que si `SaveAccount` falla, se debe intervenir manualmente; no intentar un rollback que no puede garantizarse.

## 6. Conclusión

- El fix de Edgar **responde al bug reportado**: detecta el fallo de `SaveAccount` e intenta compensar.
- La **intención es correcta**, pero la **implementación del rollback on-chain es frágil** y, en la práctica, probablemente no funcione porque la transacción de graduación ya revocó las claves custodiales que se necesitarían para firmar el rollback.
- Los checks remotos pasan, pero la cobertura de tests no alcanza a detectar el problema de fondo porque el test de rollback usa mocks.
- Se recomienda no confiar en el rollback on-chain actual y diseñar una estrategia de reconciliación DB/chain más robusta.
