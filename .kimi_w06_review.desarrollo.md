## 1. Resumen del cambio

El PR introduce el endpoint `POST /accounts/{address}/transfer` para ejecutar transferencias de certificados ArtDrop vía el contrato `ArtDropCore`, y refuerza el aislamiento de scopes entre `/transfer` (`account.transfer`) y `/sign` (`account.sign`).

Respecto a la versión original de la rama `task/W-06-transfer-sign` (`0760477`), el cambio final que llegó a `main` **migra la funcionalidad de transfer del core al plugin `artdrop/`**:

- El handler pasa de `handlers/transactions_func.go` a `artdrop/handler.go`.
- El servicio pasa de `transactions/service.go` a `artdrop/service.go`.
- El CDC pasa de `flow/cadence/transactions/protocol_transfer.cdc` a `artdrop/cdc/protocol_transfer.cdc`.
- Se añade `TxTypeTransfer` en `artdrop/types.go`.
- Se registra la ruta en `artdrop/plugin.go`.

Además, se añade un **custodial signing guard** en `transactions/service.go` y `handlers/transactions_func.go` para rechazar firmas y transacciones sobre cuentas no-custodiales.

## 3. Revisión de `artdrop/cdc/protocol_transfer.cdc`

Archivo: `artdrop/cdc/protocol_transfer.cdc` (nuevo)

```cadence
import "ArtDropCore"

transaction(certificateId: UInt64, from: Address, to: Address) {
    prepare(signer: auth(BorrowValue) &Account) {
        let protoTransfer = signer.storage
            .borrow<auth(ArtDropCore.ProtocolTransfer) &ArtDropCore.ProtocolTransferAuthority>(
                from: ArtDropCore.ProtocolTransferStoragePath
            ) ?? panic("protocol_transfer: ProtocolTransferAuthority not found - must be called from the ArtDropCore deployer account")

        protoTransfer.protocolTransfer(
            certificateId: certificateId,
            from: from,
            to: to
        )
    }
}
```

### 3.1 Entitlements

- `prepare(signer: auth(BorrowValue) &Account)` es correcto para Cadence 1.0; solo necesita `BorrowValue` para leer de `storage`.
- El tipo `auth(ArtDropCore.ProtocolTransfer) &ArtDropCore.ProtocolTransferAuthority` usa el entitlement `ProtocolTransfer` definido en `ArtDropCore`, lo cual es el patrón correcto para restringir la capacidad de transferencia.

### 3.2 Imports

- `import "ArtDropCore"` es coherente con el resto de CDCs del plugin (`setup_collection.cdc`, `register_provider.cdc`, `create_escrow.cdc`, etc.).
- **Advertencia:** este patrón de "identifier import" requiere que el entorno de ejecución/Flow SDK tenga configurado un `ContractNameResolver` que mapee `"ArtDropCore"` a la dirección de despliegue. El wallet API no realiza reemplazo de placeholders en el script (ver `transactions/service.go:buildFlowTransaction`), por lo que el despliegue/emulador debe proveer esa resolución. Esto es un riesgo operativo compartido con todo el plugin, no exclusivo de esta transacción.

### 3.3 Argumentos

- La firma `transaction(certificateId: UInt64, from: Address, to: Address)` coincide con los argumentos que construye `artdrop/service.go:Transfer`:
  1. `cadence.NewUInt64(*req.CertificateID)`
  2. `cadence.NewAddress(flow.HexToAddress(address))`  → `from`
  3. `cadence.NewAddress(flow.HexToAddress(to))`      → `to`

### 3.4 Issue de diseño: quién firma

La transacción exige que **el signer sea la cuenta que almacena `ProtocolTransferAuthority`** (normalmente el deployer de `ArtDropCore`). Sin embargo, `artdrop/service.go:Transfer` invoca:

```go
return s.deps.Transactions.Create(ctx, sync, address, string(script), args, TxTypeTransfer)
```

donde `address` es el parámetro de ruta `{address}` (el `from`). Esto significa que el **proponente/authorizer de la transacción es la cuenta `from`**, no el deployer/admin.

- En el escenario "day-1 mono-account" (deployer == from) funciona.
- En cross-account, la transacción revertirá on-chain porque `from` no posee `ProtocolTransferAuthority`.

**Recomendación:** para un protocol transfer auténticamente "admin-initiated", el servicio debería proponer la transacción con la cuenta admin (`s.deps.Config.AdminAddress`) como proponente, manteniendo `from` y `to` únicamente como argumentos de la transacción. La interfaz `transactions.Service.Create` no lo permite hoy, por lo que esto requeriría o bien extender la interfaz o construir un mecanismo específico en el plugin.

## 4. Revisión de la migración al plugin `artdrop/`

### 4.1 Handler (`artdrop/handler.go`)

- `Transfer()` usa `handlers.UseJson`, que exige `Content-Type: application/json`. Consistente con otros endpoints POST.
- Valida body vacío, JSON inválido, `to` vacío y `certificateId` ausente (`nil`).
- Soporta `?sync` igual que el resto de endpoints de transacciones.
- **OK.**

### 4.2 Service (`artdrop/service.go`)

- Lee el script desde `s.deps.Config.ScriptPathProtocolTransfer`.
- Valida `address` y `req.To` con `flow_helpers.ValidateAddress`.
- Construye 3 argumentos Cadence correctamente tipados.
- **Issue:** como se mencionó en §3.4, usa el `from` como proponente en lugar del admin.
- **Issue menor:** la validación `req.CertificateID == nil` está duplicada en handler y service.

### 4.3 Plugin registration (`artdrop/plugin.go`)

- La ruta `POST /accounts/{address}/transfer` se registra correctamente en el plugin.
- El test `TestTransferRouteBelongsToArtDropPlugin` en `main_auth_routes_test.go` verifica que no esté en el core router y que sí esté en el plugin router con scope `account.transfer`.
- **OK.**

### 4.4 Configuración (`configs/configs.go`)

```go
ScriptPathProtocolTransfer string `env:"SCRIPT_PATH_PROTOCOL_TRANSFER" envDefault:"artdrop/cdc/protocol_transfer.cdc"`
```

- Default apunta al archivo correcto dentro del repo.
- **OK.**

### 4.5 Dockerfile

```dockerfile
COPY --from=builder /build/artdrop/cdc /artdrop/cdc
```

- Se copian los CDCs del plugin a la imagen `scratch`. Sin esto, el default de `SCRIPT_PATH_PROTOCOL_TRANSFER` no funcionaría en producción.
- **OK.**

### 4.6 Tipo de transacción (`artdrop/types.go` vs `transactions/types.go`)

`artdrop/types.go` define:

```go
TxTypeTransfer transactions.Type = "ArtdropTransfer"
```

Pero el enum central en `transactions/types.go` no incluye `ArtDropTransfer`:

```go
const (
    Unknown      Type = "Unknown"
    General      Type = "General"
    FtSetup      Type = "FtSetup"
    FtTransfer   Type = "FtTransfer"
    NftSetup     Type = "NftSetup"
    NftTransfer  Type = "NftTransfer"
    ArtDropSetup Type = "ArtDropSetup"
)
```

**Issues derivados:**

1. **Inconsistencia de naming:** `ArtdropTransfer` (d minúscula) vs `ArtDropSetup` (D mayúscula).
2. **`StatusFromText` no reconoce `"ArtdropTransfer"`**. Si algún consumidor deserializa el tipo desde JSON/texto, caerá en `Unknown`.
3. La descripción del PR dice que se actualizó `transactions/types.go` y `transactions/type_string.go`, pero `transactions/type_string.go` **no existe en el repo** y `transactions/types.go` final no contiene `ArtDropTransfer`. El merge resolvió un conflicto descartando `ArtDropTransfer` del enum central.

**Recomendación:** agregar `ArtDropTransfer Type = "ArtDropTransfer"` al enum central de `transactions/types.go` y actualizar `StatusFromText` para que reconozca `"artdroptransfer"`; o, alternativamente, renombrar `TxTypeTransfer` a `"ArtDropTransfer"` y documentar que los tipos de plugin no necesitan estar en el enum central (pero esto último es frágil).

## 5. Revisión del aislamiento de scopes (/transfer vs /sign)

### 5.1 Tests de autenticación (`tests/w06_transfer_sign_test.go`)

El archivo añade 3 tests:

- `TestW06TransferAuth`: 401 sin token, 403 con scope `account.sign`, 401 token expirado, 201 con `account.transfer`.
- `TestW06SignAuth`: 401 sin token, 403 con scope `account.transfer`, 401 token expirado, 201 con `account.sign`.
- `TestW06CrossScopeIsolation`: un token `account.transfer` no puede llamar `/sign` y viceversa.

**OK.** Los tests cubren los casos de borde principales y no dependen de base de datos ni Flow emulator.

### 5.2 OpenAPI (`openapi.yml`)

- `/accounts/{address}/transfer` requiere `account.transfer`.
- `/accounts/{address}/sign` sigue requiriendo `account.sign`.
- El schema `protocolTransferRequest` está definido con `certificateId` (int64) y `to` (string).
- **OK.**

### 5.3 Custodial signing guard

- `transactions/options.go` añade `CustodialSigningGuard` y `WithCustodialSigningGuard`.
- `transactions/service.go:buildFlowTransaction` invoca el guard antes de construir la transacción.
- `handlers/transactions_func.go` invoca `s.accountService.RequireCustodialForSigning(address)` en `CreateFunc` y `SignFunc`.
- Esto cumple con el requisito de que `/sign` (y transacciones raw) sobre cuentas graduadas retornen 403.
- **OK** (nota: el guard fue introducido/fortalecido en este PR; la lógica de `RequireCustodialForSigning` vive en `accounts/signing_guard.go`).

## 8. Veredicto

**NO OK para producción.**

- La funcionalidad de `/transfer` y el aislamiento de scopes `/sign` están **bien implementadas a nivel de handler, plugin y tests de autenticación**.
- Sin embargo, existen **dos bloqueadores**:
  1. `main` no compila por culpa de W-04, por lo que no se puede validar que los tests pasen.
  2. El proponente de la transacción de protocol transfer es incorrecto para cualquier escenario que no sea mono-account (deployer == from).
- Además, la inconsistencia del tipo `ArtdropTransfer` frente al enum central es un riesgo de serialización/deserialización que debe corregirse.

**Acciones sugeridas antes de dar por cerrado W-06:**

1. Arreglar la compilación de `main` (W-04).
2. Corregir el proponente de `protocol_transfer.cdc` para usar la cuenta admin.
3. Normalizar el tipo `ArtDropTransfer` en el enum central de `transactions`.
4. Re-correr `go test ./...` y `go test ./tests/ -run "TestW06" -v` para confirmar verde.
